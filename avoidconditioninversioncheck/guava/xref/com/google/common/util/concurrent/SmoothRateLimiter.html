26  abstract class SmoothRateLimiter extends RateLimiter {
27    /*
28     * How is the RateLimiter designed, and why?
29     *
30     * The primary feature of a RateLimiter is its "stable rate", the maximum rate that
31     * is should allow at normal conditions. This is enforced by "throttling" incoming
32     * requests as needed, i.e. compute, for an incoming request, the appropriate throttle time,
33     * and make the calling thread wait as much.
34     *
35     * The simplest way to maintain a rate of QPS is to keep the timestamp of the last
36     * granted request, and ensure that (1/QPS) seconds have elapsed since then. For example,
37     * for a rate of QPS=5 (5 tokens per second), if we ensure that a request isn't granted
38     * earlier than 200ms after the last one, then we achieve the intended rate.
39     * If a request comes and the last request was granted only 100ms ago, then we wait for
40     * another 100ms. At this rate, serving 15 fresh permits (i.e. for an acquire(15) request)
41     * naturally takes 3 seconds.
42     *
43     * It is important to realize that such a RateLimiter has a very superficial memory
44     * of the past: it only remembers the last request. What if the RateLimiter was unused for
45     * a long period of time, then a request arrived and was immediately granted?
46     * This RateLimiter would immediately forget about that past underutilization. This may
47     * result in either underutilization or overflow, depending on the real world consequences
48     * of not using the expected rate.
49     *
50     * Past underutilization could mean that excess resources are available. Then, the RateLimiter
51     * should speed up for a while, to take advantage of these resources. This is important
52     * when the rate is applied to networking (limiting bandwidth), where past underutilization
53     * typically translates to "almost empty buffers", which can be filled immediately.
54     *
55     * On the other hand, past underutilization could mean that "the server responsible for
56     * handling the request has become less ready for future requests", i.e. its caches become
57     * stale, and requests become more likely to trigger expensive operations (a more extreme
58     * case of this example is when a server has just booted, and it is mostly busy with getting
59     * itself up to speed).
60     *
61     * To deal with such scenarios, we add an extra dimension, that of "past underutilization",
62     * modeled by "storedPermits" variable. This variable is zero when there is no
63     * underutilization, and it can grow up to maxStoredPermits, for sufficiently large
64     * underutilization. So, the requested permits, by an invocation acquire(permits),
65     * are served from:
66     * - stored permits (if available)
67     * - fresh permits (for any remaining permits)
68     *
69     * How this works is best explained with an example:
70     *
71     * For a RateLimiter that produces 1 token per second, every second
72     * that goes by with the RateLimiter being unused, we increase storedPermits by 1.
73     * Say we leave the RateLimiter unused for 10 seconds (i.e., we expected a request at time
74     * X, but we are at time X + 10 seconds before a request actually arrives; this is
75     * also related to the point made in the last paragraph), thus storedPermits
76     * becomes 10.0 (assuming maxStoredPermits >= 10.0). At that point, a request of acquire(3)
77     * arrives. We serve this request out of storedPermits, and reduce that to 7.0 (how this is
78     * translated to throttling time is discussed later). Immediately after, assume that an
79     * acquire(10) request arriving. We serve the request partly from storedPermits,
80     * using all the remaining 7.0 permits, and the remaining 3.0, we serve them by fresh permits
81     * produced by the rate limiter.
82     *
83     * We already know how much time it takes to serve 3 fresh permits: if the rate is
84     * "1 token per second", then this will take 3 seconds. But what does it mean to serve 7
85     * stored permits? As explained above, there is no unique answer. If we are primarily
86     * interested to deal with underutilization, then we want stored permits to be given out
87     * /faster/ than fresh ones, because underutilization = free resources for the taking.
88     * If we are primarily interested to deal with overflow, then stored permits could
89     * be given out /slower/ than fresh ones. Thus, we require a (different in each case)
90     * function that translates storedPermits to throtting time.
91     *
92     * This role is played by storedPermitsToWaitTime(double storedPermits, double permitsToTake).
93     * The underlying model is a continuous function mapping storedPermits
94     * (from 0.0 to maxStoredPermits) onto the 1/rate (i.e. intervals) that is effective at the given
95     * storedPermits. "storedPermits" essentially measure unused time; we spend unused time
96     * buying/storing permits. Rate is "permits / time", thus "1 / rate = time / permits".
97     * Thus, "1/rate" (time / permits) times "permits" gives time, i.e., integrals on this
98     * function (which is what storedPermitsToWaitTime() computes) correspond to minimum intervals
99     * between subsequent requests, for the specified number of requested permits.
100    *
101    * Here is an example of storedPermitsToWaitTime:
102    * If storedPermits == 10.0, and we want 3 permits, we take them from storedPermits,
103    * reducing them to 7.0, and compute the throttling for these as a call to
104    * storedPermitsToWaitTime(storedPermits = 10.0, permitsToTake = 3.0), which will
105    * evaluate the integral of the function from 7.0 to 10.0.
106    *
107    * Using integrals guarantees that the effect of a single acquire(3) is equivalent
108    * to { acquire(1); acquire(1); acquire(1); }, or { acquire(2); acquire(1); }, etc,
109    * since the integral of the function in [7.0, 10.0] is equivalent to the sum of the
110    * integrals of [7.0, 8.0], [8.0, 9.0], [9.0, 10.0] (and so on), no matter
111    * what the function is. This guarantees that we handle correctly requests of varying weight
112    * (permits), /no matter/ what the actual function is - so we can tweak the latter freely.
113    * (The only requirement, obviously, is that we can compute its integrals).
114    *
115    * Note well that if, for this function, we chose a horizontal line, at height of exactly
116    * (1/QPS), then the effect of the function is non-existent: we serve storedPermits at
117    * exactly the same cost as fresh ones (1/QPS is the cost for each). We use this trick later.
118    *
119    * If we pick a function that goes /below/ that horizontal line, it means that we reduce
120    * the area of the function, thus time. Thus, the RateLimiter becomes /faster/ after a
121    * period of underutilization. If, on the other hand, we pick a function that
122    * goes /above/ that horizontal line, then it means that the area (time) is increased,
123    * thus storedPermits are more costly than fresh permits, thus the RateLimiter becomes
124    * /slower/ after a period of underutilization.
125    *
126    * Last, but not least: consider a RateLimiter with rate of 1 permit per second, currently
127    * completely unused, and an expensive acquire(100) request comes. It would be nonsensical
128    * to just wait for 100 seconds, and /then/ start the actual task. Why wait without doing
129    * anything? A much better approach is to /allow/ the request right away (as if it was an
130    * acquire(1) request instead), and postpone /subsequent/ requests as needed. In this version,
131    * we allow starting the task immediately, and postpone by 100 seconds future requests,
132    * thus we allow for work to get done in the meantime instead of waiting idly.
133    *
134    * This has important consequences: it means that the RateLimiter doesn't remember the time
135    * of the _last_ request, but it remembers the (expected) time of the _next_ request. This
136    * also enables us to tell immediately (see tryAcquire(timeout)) whether a particular
137    * timeout is enough to get us to the point of the next scheduling time, since we always
138    * maintain that. And what we mean by "an unused RateLimiter" is also defined by that
139    * notion: when we observe that the "expected arrival time of the next request" is actually
140    * in the past, then the difference (now - past) is the amount of time that the RateLimiter
141    * was formally unused, and it is that amount of time which we translate to storedPermits.
142    * (We increase storedPermits with the amount of permits that would have been produced
143    * in that idle time). So, if rate == 1 permit per second, and arrivals come exactly
144    * one second after the previous, then storedPermits is _never_ increased -- we would only
145    * increase it for arrivals _later_ than the expected one second.
146    */

148   /**
149    * This implements the following function where coldInterval = coldFactor * stableInterval.
150    *
151    *          ^ throttling
152    *          |
153    *    cold  +                  /
154    * interval |                 /.
155    *          |                / .
156    *          |               /  .   <-- "warmup period" is the area of the trapezoid between
157    *          |              /   .       thresholdPermits and maxPermits
158    *          |             /    .
159    *          |            /     .
160    *          |           /      .
161    *   stable +----------/  WARM .
162    * interval |          .   UP  .
163    *          |          . PERIOD.
164    *          |          .       .
165    *        0 +----------+-------+--------------> storedPermits
166    *          0 thresholdPermits maxPermits
167    * Before going into the details of this particular function, let's keep in mind the basics:
168    * 1) The state of the RateLimiter (storedPermits) is a vertical line in this figure.
169    * 2) When the RateLimiter is not used, this goes right (up to maxPermits)
170    * 3) When the RateLimiter is used, this goes left (down to zero), since if we have storedPermits,
171    *    we serve from those first
172    * 4) When _unused_, we go right at a constant rate! The rate at which we move to
173    *    the right is chosen as maxPermits / warmupPeriod.  This ensures that the time it takes to
174    *    go from 0 to maxPermits is equal to warmupPeriod.
175    * 5) When _used_, the time it takes, as explained in the introductory class note, is
176    *    equal to the integral of our function, between X permits and X-K permits, assuming
177    *    we want to spend K saved permits.
178    *
179    *    In summary, the time it takes to move to the left (spend K permits), is equal to the
180    *    area of the function of width == K.
181    *
182    *    Assuming we have saturated demand, the time to go from maxPermits to thresholdPermits is
183    *    equal to warmupPeriod.  And the time to go from thresholdPermits to 0 is
184    *    warmupPeriod/2.  (The reason that this is warmupPeriod/2 is to maintain the behavior of
185    *    the original implementation where coldFactor was hard coded as 3.)
186    *
187    *  It remains to calculate thresholdsPermits and maxPermits.
188    *
189    *  - The time to go from thresholdPermits to 0 is equal to the integral of the function between
190    *    0 and thresholdPermits.  This is thresholdPermits * stableIntervals.  By (5) it is also
191    *    equal to warmupPeriod/2.  Therefore
192    *
193    *        thresholdPermits = 0.5 * warmupPeriod / stableInterval.
194    *
195    *  - The time to go from maxPermits to thresholdPermits is equal to the integral of the function
196    *    between thresholdPermits and maxPermits.  This is the area of the pictured trapezoid, and it
197    *    is equal to 0.5 * (stableInterval + coldInterval) * (maxPermits - thresholdPermits).  It is
198    *    also equal to warmupPeriod, so
199    *
200    *        maxPermits = thresholdPermits + 2 * warmupPeriod / (stableInterval + coldInterval).
201    */
202   static final class SmoothWarmingUp extends SmoothRateLimiter {
203     private final long warmupPeriodMicros;
204     /**
205      * The slope of the line from the stable interval (when permits == 0), to the cold interval
206      * (when permits == maxPermits)
207      */
208     private double slope;
209     private double thresholdPermits;
210     private double coldFactor;
211   
212     SmoothWarmingUp(
213         SleepingStopwatch stopwatch, long warmupPeriod, TimeUnit timeUnit, double coldFactor) {
214       super(stopwatch);
215       this.warmupPeriodMicros = timeUnit.toMicros(warmupPeriod);
216       this.coldFactor = coldFactor;
217     }
218   
219     @Override
220     void doSetRate(double permitsPerSecond, double stableIntervalMicros) {
221       double oldMaxPermits = maxPermits;
222       double coldIntervalMicros = stableIntervalMicros * coldFactor;
223       thresholdPermits = 0.5 * warmupPeriodMicros / stableIntervalMicros;
224       maxPermits = thresholdPermits
225           + 2.0 * warmupPeriodMicros / (stableIntervalMicros + coldIntervalMicros);
226       slope = (coldIntervalMicros - stableIntervalMicros) / (maxPermits - thresholdPermits);
227       if (oldMaxPermits == Double.POSITIVE_INFINITY) {
228         // if we don't special-case this, we would get storedPermits == NaN, below
229         storedPermits = 0.0;
230       } else {
231         storedPermits = (oldMaxPermits == 0.0)
232             ? maxPermits // initial state is cold
233             : storedPermits * maxPermits / oldMaxPermits;
234       }
235     }
236   
237     @Override
238     long storedPermitsToWaitTime(double storedPermits, double permitsToTake) {
239       double availablePermitsAboveThreshold = storedPermits - thresholdPermits;
240       long micros = 0;
241       // measuring the integral on the right part of the function (the climbing line)
242       if (availablePermitsAboveThreshold > 0.0) {
243         double permitsAboveThresholdToTake = min(availablePermitsAboveThreshold, permitsToTake);
244         micros = (long) (permitsAboveThresholdToTake
245             * (permitsToTime(availablePermitsAboveThreshold)
246             + permitsToTime(availablePermitsAboveThreshold - permitsAboveThresholdToTake)) / 2.0);
247         permitsToTake -= permitsAboveThresholdToTake;
248       }
249       // measuring the integral on the left part of the function (the horizontal line)
250       micros += (stableIntervalMicros * permitsToTake);
251       return micros;
252     }
253   
254     private double permitsToTime(double permits) {
255       return stableIntervalMicros + permits * slope;
256     }
257 
258     @Override
259     double coolDownIntervalMicros() {
260       return warmupPeriodMicros / maxPermits;
261     }
262   }

264   /**
265    * This implements a "bursty" RateLimiter, where storedPermits are translated to
266    * zero throttling. The maximum number of permits that can be saved (when the RateLimiter is
267    * unused) is defined in terms of time, in this sense: if a RateLimiter is 2qps, and this
268    * time is specified as 10 seconds, we can save up to 2 * 10 = 20 permits. 
269    */
270   static final class SmoothBursty extends SmoothRateLimiter {
271     /** The work (permits) of how many seconds can be saved up if this RateLimiter is unused? */
272     final double maxBurstSeconds; 
273     
274     SmoothBursty(SleepingStopwatch stopwatch, double maxBurstSeconds) {
275       super(stopwatch);
276       this.maxBurstSeconds = maxBurstSeconds;
277     }
278   
279     @Override
280     void doSetRate(double permitsPerSecond, double stableIntervalMicros) {
281       double oldMaxPermits = this.maxPermits;
282       maxPermits = maxBurstSeconds * permitsPerSecond;
283       if (oldMaxPermits == Double.POSITIVE_INFINITY) {
284         // if we don't special-case this, we would get storedPermits == NaN, below
285         storedPermits = maxPermits;
286       } else {
287         storedPermits = (oldMaxPermits == 0.0)
288             ? 0.0 // initial state
289             : storedPermits * maxPermits / oldMaxPermits;
290       }
291     }
292   
293     @Override
294     long storedPermitsToWaitTime(double storedPermits, double permitsToTake) {
295       return 0L;
296     }
297 
298     @Override
299     double coolDownIntervalMicros() {
300       return stableIntervalMicros;
301     }
302   }

332   final void doSetRate(double permitsPerSecond, long nowMicros) {
333     resync(nowMicros);
334     double stableIntervalMicros = SECONDS.toMicros(1L) / permitsPerSecond;

351   @Override
352   final long reserveEarliestAvailable(int requiredPermits, long nowMicros) {
353     resync(nowMicros);
354     long returnValue = nextFreeTicketMicros;
355     double storedPermitsToSpend = min(requiredPermits, this.storedPermits);
356     double freshPermits = requiredPermits - storedPermitsToSpend;
357     long waitMicros = storedPermitsToWaitTime(this.storedPermits, storedPermitsToSpend)
358         + (long) (freshPermits * stableIntervalMicros);
359 
360     try {
361       this.nextFreeTicketMicros = LongMath.checkedAdd(nextFreeTicketMicros, waitMicros);
362     } catch (ArithmeticException e) {
363       this.nextFreeTicketMicros = Long.MAX_VALUE;
364     }
365     this.storedPermits -= storedPermitsToSpend;
366     return returnValue;
367   }

369   /**
370    * Translates a specified portion of our currently stored permits which we want to
371    * spend/acquire, into a throttling time. Conceptually, this evaluates the integral
372    * of the underlying function we use, for the range of
373    * [(storedPermits - permitsToTake), storedPermits].
374    *
375    * <p>This always holds: {@code 0 <= permitsToTake <= storedPermits}
376    */
377   abstract long storedPermitsToWaitTime(double storedPermits, double permitsToTake);

380    * Returns the number of microseconds during cool down that we have to wait to get a new permit.

384   private void resync(long nowMicros) {
385     // if nextFreeTicket is in the past, resync to now
386     if (nowMicros > nextFreeTicketMicros) {
387       storedPermits = min(maxPermits,
388           storedPermits
389             + (nowMicros - nextFreeTicketMicros) / coolDownIntervalMicros());
390       nextFreeTicketMicros = nowMicros;
391     }
